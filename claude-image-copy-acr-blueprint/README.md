# Chainguard Image Copy — Azure Bicep Deployment

Deploys Azure infrastructure to build and run `cgr-image-copy:v1` from this repo, with the Chainguard OIDC token supplied by the user at deployment time, stored in Azure Key Vault, and injected into the app as `OIDC_TOKEN`. The ACR image build is **triggered automatically** during deployment.

---

## Resources Deployed

| Resource | Description |
|---|---|
| **Azure Container Registry (ACR)** | Stores the built container image |
| **ACR Task** | Builds `cgr-image-copy:v1` from the Dockerfile in this repo |
| **Deployment Script (AzureCLI)** | Automatically runs `az acr task run` during deployment and waits for success |
| **Script Runner Identity** | Dedicated managed identity scoped to ACR Contributor, used only by the deployment script |
| **Azure Key Vault** | Holds the user-supplied `chainguard-oidc-token` secret |
| **App Managed Identity** | Grants the Container App permission to pull and push to ACR and read the Key Vault secret |
| **Container App Environment** | Managed environment for the Container App |
| **Azure Container App** | Runs the container; starts only after the build script confirms the image exists |

---

## Deployment Order (enforced by dependsOn)

```
ACR + Key Vault + Managed Identities
     ↓
Role Assignments (ACR Pull/Push, Key Vault Secrets User, ACR Contributor for script)
     ↓
ACR Task (definition)
     ↓
Deployment Script  ← triggers az acr task run, polls to completion
     ↓
Container App  ← guaranteed the image exists before first pull
```

## Using the Deploy to Azure Button

The deployment is now defined as native Bicep in `main.bicep`, with example parameters in `main.bicepparam`.

The template accepts:

| Parameter | Required | Default |
|---|---|---|
| `containerRegistryName` | **Yes** | _(none)_ |
| `targetSubscriptionId` | No | current deployment subscription |
| `acrResourceGroupName` | No | empty |
| `location` | No | Resource group location |
| `keyVaultName` | No | `kv-cgr-<uniqueSuffix>` |
| `containerAppName` | No | `cgr-image-copy-app` |
| `containerAppEnvironmentName` | No | `cgr-container-env` |
| `gitRepoUrl` | No | `https://github.com/justinprince/platform-examples` |
| `gitRepoBranch` | No | `main` |
| `dockerfilePath` | No | `Dockerfile` |
| `gitRepoContextPath` | No | `claude-image-copy-acr-blueprint` |
| `chainguardOidcToken` | **Yes** | _(none — generate with `chainctl` before deployment)_ |
| `apiEndpoint` | No | `https://console-api.enforce.dev` |
| `issuerUrl` | No | `https://issuer.enforce.dev` |
| `chainguardOrg` | No | `justin.prince` |
| `chainguardGroup` | No | `justin.prince` |
| `chainguardIdentity` | No | `taco` |
| `ignoreReferrers` | No | `true` |
| `verifySignatures` | No | `false` |

You must provide an ACR name and a Chainguard OIDC token. The token should be generated by the deploying user with `chainctl` before starting deployment.

If `acrResourceGroupName` is populated, the deployment treats `containerRegistryName` as an existing ACR and skips ACR creation. If `acrResourceGroupName` is left empty, the deployment creates the ACR in the current deployment resource group and then applies the ACR-scoped role assignments there.

> **Permissions note:** The account deploying the template must have permission to create role assignments on the resource group (Owner or User Access Administrator).

## Generate the OIDC Token

Generate the token before deployment with your normal `chainctl` flow, then provide it as the `chainguardOidcToken` parameter in the Azure Portal or CLI.

The deployment stores that value in Key Vault as `chainguard-oidc-token` and the Container App injects it into the runtime as:

```text
OIDC_TOKEN=<value>
```

## CLI Deployment

```bash
az group create --name <your-resource-group> --location eastus

az deployment group create \
  --resource-group <your-resource-group> \
  --template-file main.bicep \
  --parameters @main.bicepparam \
  --parameters targetSubscriptionId="$(az account show --query id -o tsv)" \
  --parameters acrResourceGroupName="<existing-acr-resource-group>" \
  --parameters containerRegistryName=<your-unique-acr-name> \
  --parameters chainguardOidcToken="<your-chainctl-token>"
```

Deployment takes 10–20 minutes depending on Docker build time.

## Runtime Configuration

The Container App now sets the environment required by `main.go`, including:

- `OIDC_TOKEN` from Key Vault
- `API_ENDPOINT`
- `ISSUER_URL`
- `GROUP_NAME`
- `GROUP`
- `IDENTITY`
- `DST_REPO`
- `ACR_REGISTRY`
- `IGNORE_REFERRERS`
- `VERIFY_SIGNATURES`
- `PORT`

`DST_REPO` is derived automatically as `<acr-name>.azurecr.io/cgr-image-copy`.

If you rotate the secret later, restart the Container App revision so the updated Key Vault value is picked up.

---

## RBAC Expectations

The deployment grants:

- the **app managed identity** `AcrPull`, `AcrPush`, and Key Vault Secrets User
- the **script runner identity** Contributor on the ACR so it can trigger the build task

Only the ACR-related assignments use `targetSubscriptionId` and `acrResourceGroupName` when an existing ACR is supplied. Key Vault access remains scoped to the current deployment resource group.

## Private Repo Note

This repo is public, so ACR Tasks can clone it natively. If you fork this into a private repo, add a PAT credential to the ACR Task after deployment:

```bash
az acr task credential add \
  --registry <acr-name> \
  --name build-cgr-image-copy \
  --login-server github.com \
  --username YOUR_GITHUB_USERNAME \
  --password YOUR_PAT
```
